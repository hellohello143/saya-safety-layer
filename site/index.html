Payment safety layer · Base + Solana

Your agent can spend.
It can't overspend.

Saya issues each AI agent a scoped session key with a spending cap the blockchain itself enforces — not a server that can be bypassed. Your agent pays for APIs, compute, data, or other agents over x402 in USDC. A bug, a jailbreak, or a runaway loop can never spend past the cap you set.

The problem

An API key with a card behind it is one infinite loop from an unbounded bill.

The usual way to let an agent pay is to hand it a funded wallet, or a key with a card attached. That works right up until a prompt injection, a bad tool call, or a loop that doesn't stop — and then the only thing between the agent and your balance is hope.

Saya puts a hard ceiling in between. The ceiling lives on-chain, so it holds even when the agent, the prompt, or this backend goes wrong. The agent transacts freely inside the sandbox you define, and cannot step outside it.

How it works

Issue a key. Let the agent pay. The chain does the enforcing.

1 Issue

Mint a scoped key

Set a total budget, an expiry, and per-payment limits. Saya mints it on-chain — a Spend Permission on Base, an SPL token delegation on Solana.

2 Pay

Agent hits a paywall

Your agent requests a resource, gets an x402 challenge, and asks Saya to settle it. Saya checks the policy, pulls only the capped amount, and pays the merchant in USDC.

3 Enforce

The chain caps the pull

Go over budget or past expiry and the transfer reverts on-chain — no server in the loop to bypass. Every attempt, approved or blocked, lands in the audit log.

On-chain — the chain reverts any violation, independent of Saya. Scheduled revoke — Solana has no native expiry, so Saya revokes the delegation on-chain at expiry. Policy — Saya's engine enforces it, and flags it on every payment.

Two chains, one guarantee

Native to Base and Solana — not the lowest common denominator.

Base
EVM · CDP Spend Permissions

Budget and expiry live in the Spend Permission itself. The contract reverts any spend over the cap or past the end — Saya never has to be trusted for those.

  • +Gasless user ops via the CDP paymaster
  • +Lifetime cap in a single non-resetting window
  • +Real revocation with an on-chain tx hash
Solana
SPL Token delegation

A single-hop delegation caps the total on-chain and decrements per transfer. Solana has no native time bound, so Saya revokes the delegation on-chain at expiry — actively, not just by refusing to sign.

  • +On-chain total cap via delegated amount
  • +Scheduled revoke clears the delegate at expiry
  • +Single-hop — funds never sit with the spender

What you get

A policy and audit layer on top of the on-chain cap.

Token auth

Every API route needs a bearer token. On mainnet the server refuses to start without one.

Full audit log

Every decision — approved or blocked — is recorded with a reason code and queryable.

Circuit breaker

Too many attempts in a window trips the key: auto-suspend, then revoke it on-chain.

Ops dashboard

Watch budgets burn down, see risk flags, and revoke any key with one click.

x402-native

Agents pay per request through the x402 challenge, settled with a real on-chain transfer.

USDC, done right

Six-decimal integer money end to end. No floating-point rounding, ever.
