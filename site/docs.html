Overview

Autonomous agents increasingly need to pay for things — APIs, compute, data, other agents. Hand one a funded wallet or a key with a card behind it, and you are one prompt injection or one runaway loop away from an unbounded bill.

Saya sits between the agent and the money. You issue it a session key scoped to a total budget, an expiry, and per-payment limits. The cap is minted on-chain — a CDP Spend Permission on Base, an SPL token delegation on Solana — so a bug, a jailbreak, or a compromised backend can never spend past it. Saya itself is a policy, settlement, and audit layer on top of that on-chain guarantee, not the last line of defense.

Agents pay for HTTP resources over the x402 protocol, settled in USDC. Everything runs behind one HTTP API and a dashboard.

Concepts

TermMeaning
Session keyThe scoped, on-chain spending grant you issue an agent — a budget + expiry + per-payment limits. The unit everything revolves around.
TreasuryThe account that holds USDC. Funds are pulled from here, capped by the session key.
SpenderThe account Saya uses to pull the capped funds and pay merchants. On Solana it is the delegate; on Base it is the Spend Permission's spender.
Spend PermissionThe Base (EVM) primitive: a signed grant the SpendPermissionManager contract enforces — it reverts any spend over the allowance or past the end.
DelegationThe Solana primitive: an SPL ApproveChecked that caps a delegate's spend on-chain and decrements per transfer.
x402The HTTP payment protocol: a server answers 402 Payment Required with what it wants; the client pays and retries.
Base unitsUSDC has 6 decimals. Amounts are integers internally (no floats); the API takes and returns decimal strings like "1.00".

Quickstart

Prerequisites: Node.js 22.13 or newer, and CDP credentials (see below). No native build tools required.

Local

bash
git clone https://github.com/hellohello143/saya-safety-layer.git
cd saya-safety-layer
npm install
cp .env.example .env     # then add your 3 CDP_* values
npm run setup            # validates creds, makes an API token, funds a test wallet
npm run dev              # backend + dashboard on http://127.0.0.1:3000

Docker

bash
cp .env.example .env
docker compose run --rm backend npm run setup
docker compose up        # backend + dashboard :3000, mock seller :4021

Open http://localhost:3000 for the dashboard. To watch the whole flow end to end against a local mock seller:

bash
npm run mock-seller                        # a local x402 resource
npm run agent-sim -- --network=solana-devnet  # within / over-cap / rate-limit scenarios

CDP credentials

Saya mints and moves funds through the Coinbase Developer Platform. You need three values, all from one CDP project:

  • Sign in at the CDP Portal.
  • Create a Secret API Key — this gives you CDP_API_KEY_ID and CDP_API_KEY_SECRET.
  • Create or note your Wallet Secret — this is CDP_WALLET_SECRET, and it authorizes wallet operations.

Put all three in .env. Saya creates the owner, treasury, and spender accounts by name on first use.

The API key and the Wallet Secret must belong to the same CDP project. A mismatch is the most common setup error — npm run setup catches it and tells you.

Your first payment

The whole loop in four calls. Assume the backend is running and $TOKEN holds your API token.

1 — Issue a scoped key

bash
curl -s localhost:3000/api/sessions \
  -H "authorization: Bearer $TOKEN" -H "content-type: application/json" \
  -d '{ "agentId":"scraper-07", "network":"solana-devnet",
        "maxAmountTotal":"2.00", "maxAmountPerTx":"0.10",
        "expiresAt":1751500000 }'
# -> 201 { "id": "a1b2…", "status": "active", "remainingBudget": "2.00", … }

2 — Pay for a resource

bash
curl -s localhost:3000/api/pay \
  -H "authorization: Bearer $TOKEN" -H "content-type: application/json" \
  -d '{ "sessionId":"a1b2…", "targetUrl":"http://127.0.0.1:4021/resource" }'
# -> 200 { "status": "approved", "txHash": "…", "resource": { … } }

3 — Read the audit trail

bash
curl -s "localhost:3000/api/audit?agentId=scraper-07" -H "authorization: Bearer $TOKEN"
# every decision, approved or blocked, with a reason code

4 — Revoke on-chain when you're done

bash
curl -s -X POST localhost:3000/api/sessions/a1b2…/revoke -H "authorization: Bearer $TOKEN"
# real on-chain revocation; the delegate/permission is cleared

Session keys

A session key is the scoped, on-chain wallet you hand an agent. Create one with POST /api/sessions. Amounts are decimal USDC strings; expiresAt is unix seconds; network is optional and defaults to your configured EVM network.

POST /api/sessions — body
{
  "agentId": "scraper-07",
  "network": "solana-devnet",   // optional
  "maxAmountTotal": "2.00",      // on-chain cap
  "maxAmountPerTx": "0.10",      // policy cap, per payment
  "expiresAt": 1751500000,
  "allowedRecipients": []        // empty = any recipient (flagged higher-risk)
}
  • maxAmountTotal is minted on-chain — the chain rejects any spend beyond it, on both chains.
  • maxAmountPerTx and allowedRecipients are enforced by Saya's policy engine and flagged on every payment (see the trust boundary).
  • expiresAt is enforced by the chain on Base; on Solana, Saya revokes the delegation on-chain at expiry.

Solana allows one delegate per token account, so this first cut permits one active Solana session at a time. Revoke the current one (or let it expire) before issuing another. Base has no such limit.

Paying for resources

Point a session at a paywalled URL and Saya runs the whole x402 exchange for you:

  • Fetches targetUrl. If it isn't 402, the body is returned as-is (no spend).
  • On 402, parses the payment requirements and runs the policy checks.
  • Pulls only the capped amount on-chain, then settles the merchant in USDC and retries with the payment proof.
  • Records the attempt — approved or blocked — in the audit log.
POST /api/pay
// request
{ "sessionId": "a1b2…", "targetUrl": "https://api.example.com/data" }

// approved -> 200
{ "status": "approved", "txHash": "…", "resource": { … }, "riskFlags": [] }

// blocked by a limit -> 402
{ "status": "rejected_policy", "reason": "EXCEEDS_TOTAL_LIMIT" }

The fund flow differs by chain: Base is two-hop (pull to the spender, then pay the merchant); Solana is single-hop (the delegate pays the merchant directly, so funds never sit with the spender).

Funding

The treasury must hold USDC to make payments. npm run setup prints the treasury address and, on testnet, requests test USDC from a faucet for you.

  • Base — smart-account operations are gasless via the CDP paymaster, so you only need USDC (no test ETH). Testnet USDC comes from the CDP faucet or Circle's faucet.
  • Solana — payments are not gasless: the treasury needs a little SOL for fees. On devnet, setup requests both SOL and USDC.
  • Mainnet — there is no faucet. setup tells you which address to fund with real USDC (and SOL, on Solana).

Operations

  • Circuit breaker — too many attempts in the window trips the key: Saya auto-suspends it, then revokes it on-chain. Tune with CIRCUIT_BREAKER_MAX_ATTEMPTS and CIRCUIT_BREAKER_WINDOW_SECONDS. It counts every attempt, not just successes.
  • Audit log — every decision, approved or blocked, recorded with a reason code and queryable by agent, session, decision, on-chain status, or time.
  • Dashboard — at /: create sessions, watch budgets burn down, see risk flags, and revoke any key on-chain with one click. It asks for the API token once.

Authentication

Every route under /api/ requires the API token. Send it as a bearer token (or the X-API-Key header); it is compared in constant time.

http
Authorization: Bearer <API_TOKEN>

The static dashboard and GET /health stay open. On any mainnet network the server is fail-closed: it will not start without API_TOKEN, so real funds are never served unprotected.

Returned by the session endpoints. Amounts are decimal USDC strings; timestamps are unix seconds.

json

  "agentId": "scraper-07",

  "network": "solana-devnet",
  "expiryEnforcedOnChain": false,   // true on Base, false on Solana

  "maxAmountTotal": "2.00",

  "allowedRecipients": [],
  "higherRisk": true,             // no recipient allowlist

Add ?onchain=true to GET /api/sessions/:id to attach a live onchain read (validity, revocation, remaining allowance) straight from the chain rather than the database.

Errors & status codes

A blocked payment is not an exception — it is a normal response with a machine-readable reason. HTTP status maps to the outcome:

StatusMeaning
200OK — payment approved, or a list/detail read.
201Session key issued.
400invalid_request — the body failed validation.
401unauthorized — missing or wrong API token.
402rejected_policy — a limit blocked the payment (see the reason).
404not_found — no such session.
409Conflict — e.g. an active Solana session already exists, or nothing to revoke.
502onchain_error / rejected_onchain — an on-chain call failed.

On mainnet, raw chain error messages are logged server-side and redacted from the response (they can contain RPC URLs). On testnet the detail field carries them for debugging.

A rejected payment always carries one of these in its reason, so your agent can branch on it:

Risk flags

Approved payments can still carry riskFlags — things that succeeded but are worth watching. They also land on the audit row.

FlagMeaning
no_allowlist_sessionThe session has no recipient allowlist, so it can pay anyone.
settlement_uncertainAn ambiguous on-chain failure; the cap is treated as consumed to avoid a double-pay.
settlement_failedThe merchant-payment leg failed after funds were pulled (Base two-hop).
resource_not_deliveredPayment settled, but the seller didn't return the resource.
target_error_statusThe target returned a non-2xx before any payment.
onchain_revoke_failedThe breaker suspended the key, but the on-chain revoke didn't confirm.

Trust boundary

The whole point of Saya is being honest about where each limit is enforced. No limit is ever silently downgraded to something weaker.

LimitSolanaBase
Total budgetOn-chainOn-chain
ExpiryScheduled revokeOn-chain
RevocationOn-chainOn-chain
Per-payment capPolicyPolicy
Recipient allowlistPolicyPolicy
  • On-chain — the chain reverts any violation, independent of Saya.
  • Scheduled revoke — Solana has no native expiry, so Saya issues an on-chain Revoke at expiry (and sweeps any that expired while it was down). Backend-driven, not protocol-intrinsic.
  • Policy — enforced by Saya's engine and surfaced as a risk flag on every payment.

The full, primary-source-cited breakdown lives in TRUST_BOUNDARY.md.

Chains

Solana — SPL Token delegation

A single-hop delegation caps the total on-chain via delegated_amount and decrements it per transfer, so funds never sit with the spender. Solana has no native time bound, so Saya revokes the delegation on-chain at expiry — actively, not just by refusing to sign. Payments need a little SOL for fees, and one Solana session is active at a time.

Base — CDP Spend Permissions

Budget and expiry live in the Spend Permission itself; the SpendPermissionManager contract reverts any spend over the cap or past the end. Smart-account operations are gasless via the CDP paymaster, so you only need USDC. The lifetime cap is a single non-resetting window, and revocation records a real on-chain transaction hash.

Deployment & scaling

Saya runs as a single process: SQLite for storage and an in-process timer for the Solana expiry sweep. That is the intended shape — one operator, one instance — and it keeps spend accounting race-free without external infrastructure.

  • Docker — docker compose up runs the backend, dashboard, and mock seller; a named volume persists the database.
  • State — the circuit-breaker counts are DB-backed, so they survive a restart.
  • Clustering — to run multiple instances, swap the repository layer (src/db/) to Postgres and elect one leader for the expiry sweep. Until then, run one instance.

Security

Saya ships with token auth and mainnet guardrails, and its payment flows were verified live. Before pointing it at real funds:

  • Terminate TLS in front of it — the token is a bearer credential; never send it over plain HTTP.
  • Keep the port off the open internet — bind to 127.0.0.1 (the default) and reach it through a proxy.
  • Store secrets in a real secret manager — the CDP_* values, the wallet secret, and API_TOKEN.
  • Fund deliberately — the on-chain caps bound the spend; a modest treasury float bounds the blast radius.

It has not had a third-party security audit — review the trust boundary before trusting it with significant funds.

Troubleshooting

The server won't start on mainnet

By design. With NETWORK=base or SOLANA_NETWORK=solana set, Saya refuses to start without API_TOKEN. Set one (or run npm run setup) and restart.

"Project has no secret" / credential errors

Your API key and Wallet Secret belong to different CDP projects. Generate the Wallet Secret with the same project selected, and re-copy both into .env with no quotes or spaces around =.

A Solana session won't create (409)

An active Solana session already exists — SPL allows one delegate per token account. Revoke it (or wait for expiry) before issuing another.

Payments to my local mock seller are rejected

On mainnet, loopback and private hosts are blocked as an SSRF guard, so a local mock seller only works on testnet. On testnet, 127.0.0.1 is allowed.
